## The lean process (detail in `delivery_process.md`)
- Model: **`AT → Topic → component`**. A **Topic** = the delivery unit (one `topic_NN/` folder; the level scheduled into sessions; matches the Delivery Plan template's "Topic" field). Components (C1, C2…) are derived **straight from the AT** — no separate decomposition layer.
- Two docs per Topic: **`coverage.md`** (UoC + AT-alignment spec; only UoC/AT cross-refs) + **`slide_plan.md`** (deck build sheet — teach + exercise slides interleaved in deck order; `[BESPOKE]` = author from inline brief · `[AWS Mx Sy]` = AWS slide to reuse · `[EX]` = exercise). The plan **identifies the AWS slides the Topic needs** up front.
- **The AT sets the depth ceiling** (e.g. Topic 1 = exactly Business Case Appendix 2 Q1–Q5, recognise/explain/classify, not build).
- **Settled slide-creation flow (2026-06-01; isolation step dropped 2026-06-02):** `coverage.md` → `slide_plan.md` (pins the exact AWS deck+slide refs) → **(agent)** generates the **Kangan-branded** deck via a per-Topic script (`scripts/build_kangan_topicNN_deck.py`), reading the pinned AWS slides **directly from `original-materials/…`** and authoring **every** slide — bespoke *and* AWS-sourced — fresh into the brand layouts with **labelled placeholders** for images → **(human)** pastes the real AWS images **straight from the main AWS deck folder**. **No `source_slides/` isolation step** — it didn't assist (the slide_plan pin table + on-slide markers carry provenance). Authored-with-placeholders beats re-skinning (which mis-sizes text + drops vector diagrams).
- **Marking the AWS source on a slide:** image/diagram from AWS → a **labelled image placeholder** naming the ref (e.g. "AWS — VPC diagram (ACA M07 S__)"); content/text-only AWS slide (no image) → author reuse-first **and** add a visible on-slide note **"take from AWS [deck · slide ref]"**. (Tim, 2026-06-02.) **AWS source refs MAY remain visible to students** (kicker provenance, placeholder labels, demo `source=`) — attribution to reused AWS material, a sanctioned exception to the in-world-slide rule (Tim, 2026-06-03).
- **Reuse vs rewrite (open thread, 2026-06-03):** Tim flagged that authoring AWS content *fresh into Kangan slides* is closer to **rewriting** than reusing. Topic 8 was resolved by Tim **consolidating** my drafted Kangan slides **with the actual AWS reference slides** into a complete deck (hence 8a 42 / 8b 33 slides). The pure-reuse-vs-rewrite model for Topics 9–14 (and whether to retrofit 6/7) is **not yet settled** — revisit.
- **Primer-first (no assumed baseline):** every technical concept is taught fundamentals-first — a short vendor-neutral **`[PRIMER]`** (what a subnet is, how DNS works, what a firewall does) **before** the AWS-context slide. Per-concept: `[PRIMER] → [AWS] teach → [DEMO] → [EX]`. Reuse-first still applies (pin an AWS "basics" slide where one exists; bespoke primer only for gaps). (Tim, 2026-06-02 — applied from Topic 7.)
- **Reuse-first (AWS-heavy topics) — governing rule:** where an AWS deck covers a teach point, author the slide **FROM the AWS slide's actual content** — extract + read the relevant AWS module(s) from `original-materials/AWS-Instructor Presentations/` **first**; bespoke only for genuine gaps. Do **not** approximate from your own knowledge when AWS covers it (the mistake that prompted this rule). ACA decks need the cp-to-short-temp-path trick to open.
- **AWS practicals: `teach → demonstrate → practice`** — a `[DEMO]` sits before the activity. **Demos use an AWS recorded demo wherever a suitable one exists** (catalogue: `planning/aws-recorded-demos-catalogue.md`); a **live instructor demo is the fallback only** when none exists. Non-AWS/analytical activities keep plain `teach → practice`. (`demo_slide` layout in `kangan_deck.py`, `source` param = recorded-demo ref.)
- **Brand:** teaching decks wear **Kangan/BKI** branding (gold `#EDAB0C` + charcoal, Roboto), **not** the in-world YAT case-study brand — see `documentaion/kangan-branding.md`. (YAT brand stays inside the scenario documents.)
- **Git-tracking decision (2026-06-01):** the **assembled, curated Topic decks (`topic_NN/Topic_NN_Slides.pptx`) ARE git-tracked** — the repo is org-owned and the **primary instructor-to-instructor channel** for finished teaching materials, so the canonical deck must live there. Only the **raw AWS source decks (`**/source_slides/`) + `/original-materials/` stay git-ignored** — reason is **size + file count** (1 GB+, many files), *not* licensing (audience is authorised instructors, covered by reuse rights). `.pptx` binary bloat acceptable for now; Git LFS is the fallback if history grows.

## Working mode (deck generation)
The deck is **generated** from `slide_plan.md` by the per-Topic Kangan build script — the agent authors the full deck (bespoke + AWS-sourced slides, images as placeholders); the human isolates the AWS slides beforehand and pastes the real images in afterward. `slide_plan.md` is **disposable scaffold** — deleted once the deck is built; the **deck (`.pptx`) is the source of truth**. (The earlier Topic-1 mode of drafting slide copy on-demand in chat is superseded by script generation.) Same spirit as [[feedback-process-docs-current-only]] / [[draft-naming-pragmatism]].
